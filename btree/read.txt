<h1>read(2) - Linux man page</h1>

<h2>Name</h2>
read - read from a file descriptor
<h2>Synopsis</h2>
<pre class="code">#include &lt;<a href="/include/unistd.h" rel="nofollow">unistd.h</a>&gt;
ssize_t read(int fd, void *buf, size_t count);</pre>
<h2>Description</h2>

<b>read</b>() attempts to read up to <i>count</i> bytes from file descriptor <i>fd</i> into the
buffer starting at <i>buf</i>.
<p>On files that support seeking, the read operation commences at the current file offset, and the file offset is incremented by the number of bytes read. If
the current file offset is at or past the end of file, no bytes are read, and <b>read</b>() returns zero.
<p>If <i>count</i> is zero, <b>read</b>() <i>may</i> detect the errors described below. In the absence of any errors, or if <b>read</b>() does not check for
errors, a <b>read</b>() with a <i>count</i> of 0 returns zero and has no other effects.
<p>If <i>count</i> is greater than <b>SSIZE_MAX</b>, the result is unspecified.
<h2>Return Value</h2>
On success, the number of bytes read is returned (zero indicates end of file), and the file
position is advanced by this number. It is not an error if this number is smaller than the number of bytes requested; this may happen for example because fewer
bytes are actually available right now (maybe because we were close to end-of-file, or because we are reading from a pipe, or from a terminal), or because
<b>read</b>() was interrupted by a signal. On error, -1 is returned, and <i>errno</i> is set appropriately. In this case it is left unspecified whether the
file position (if any) changes.
<h2>Errors</h2>
<dl compact>
<dt><b>EAGAIN</b>
<dd>The file descriptor <i>fd</i> refers to a file other than a socket and has been marked nonblocking (<b>O_NONBLOCK</b>), and the read would block.
<dt><b>EAGAIN</b> or <b>EWOULDBLOCK</b>
<dd>The file descriptor <i>fd</i> refers to a socket and has been marked nonblocking (<b>O_NONBLOCK</b>), and the read would block. POSIX.1-2001 allows either
error to be returned for this case, and does not require these constants to have the same value, so a portable application should check for both possibilities.
<dt><b>EBADF</b>
<dd><i>fd</i> is not a valid file descriptor or is not open for reading.
<dt><b>EFAULT</b>
<dd><i>buf</i> is outside your accessible address space.
<dt><b>EINTR</b>
<dd>The call was interrupted by a signal before any data was read; see <i><b><a href="/man/7/signal">signal</a></b>(7)</i>.
<dt><b>EINVAL</b>
<dd><i>fd</i> is attached to an object which is unsuitable for reading; or the file was opened with the <b>O_DIRECT</b> flag, and either the address specified
in <i>buf</i>, the value specified in <i>count</i>, or the current file offset is not suitably aligned.
<dt><b>EINVAL</b>
<dd><i>fd</i> was created via a call to <i><b><a href="timerfd_create">timerfd_create</a></b>(2)</i> and the wrong size buffer was given to <b>read</b>(); see
<i><b><a href="timerfd_create" rel="nofollow">timerfd_create</a></b>(2)</i> for further information.
<dt><b>EIO</b>
<dd>I/O error. This will happen for example when the process is in a background process group, tries to read from its controlling terminal, and either it is
ignoring or blocking <b>SIGTTIN</b> or its process group is orphaned. It may also occur when there is a low-level I/O error while reading from a disk or tape.
<dt><b>EISDIR</b>
<dd><i>fd</i> refers to a directory.</dl>
<p>Other errors may occur, depending on the object connected to <i>fd</i>. POSIX allows a <b>read</b>() that is interrupted after reading some data to return
-1 (with <i>errno</i> set to <b>EINTR</b>) or to return the number of bytes already read.
<h2>Conforming to</h2>
SVr4, 4.3BSD, POSIX.1-2001.
<h2>Notes</h2>
On NFS file systems, reading small amounts of data will only update the timestamp the first time,
subsequent calls may not do so. This is caused by client side attribute caching, because most if not all NFS clients leave st_atime (last file access time)
updates to the server and client side reads satisfied from the client's cache will not cause st_atime updates on the server as there are no server side reads.
UNIX semantics can be obtained by disabling client side attribute caching, but in most situations this will substantially increase server load and decrease
performance.
